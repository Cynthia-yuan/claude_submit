## 故障描述

### 定义
网络往返时间(RTT)异常增大，超过正常基线值，导致应用响应变慢、吞吐量下降。

### 典型场景
- 跨国/跨洋传输
- 卫星链路
- 拥塞的网络链路
- 路由绕行
- 低带宽链路

### 影响范围
- **用户体验**: 响应时间变长，操作卡顿
- **吞吐量**: TCP连接吞吐量受限
- **实时应用**: 视频会议、在线游戏等实时性要求高的应用严重影响
- **影响层级**: 应用层、传输层

## 故障表现

### 症状
- 应用响应时间明显增加（从毫秒级到秒级）
- `ping` 命令显示 RTT > 100ms（正常通常 < 10ms）
- TCP连接建立缓慢（三次握手时间长）
- 文件传输速度慢
- 视频会议卡顿、声音延迟

## 根本原因

### 常见原因
1. **网络拥塞**: 链路带宽不足，队列堆积
2. **路由问题**: 路径绕行，经过更多跳数
3. **物理距离**: 跨国、跨洋传输
4. **设备性能**: 路由器/交换机处理能力不足
5. **链路质量**: 无线干扰、光纤衰减
6. **配置问题**: QoS配置不当、流量整形

### 触发条件
- 链路利用率 > 80%
- 路由表更新导致路径变化
- 网络设备重启或切换
- 大规模数据传输占用带宽

## 检测方法

### 主动检测
```bash
# 1. Ping 测试
ping -c 100 target_host | tail -1
# 输出: min/avg/max/stddev

# 2. MTR (网络诊断工具)
mtr -r -c 100 target_host
# 显示每一跳的延迟和丢包

# 3. traceroute
traceroute target_host
# 显示路径和每一跳的延迟

# 4. tcpping (TCP ping)
tcpping -x 100 target_host 80
# 测量TCP连接建立时间

# 5. hping3
hping3 -p 80 -S target_host -c 100
# 精确测量TCP握手延迟
```

## 故障注入

### 注入方法
使用 `tc` (traffic control) 和 `netem` (network emulator) 模拟网络延迟

### 注入工具
```bash
#!/bin/bash
# 网络延迟注入脚本

# 1. 安装工具
sudo apt-get install -y iproute2

# 2. 添加延迟到指定网络接口
# 参数说明:
#   delay 200ms     - 基础延迟200毫秒
#   50ms           - 延迟抖动±50毫秒
#   25%            - 25%的相关性(延迟相关性)
#   distribution normal - 使用正态分布

# 场景1: 固定延迟
sudo tc qdisc add dev eth0 root netem delay 200ms

# 场景2: 延迟 + 抖动
sudo tc qdisc add dev eth0 root netem delay 200ms 50ms 25%

# 场景3: 延迟 + 丢包
sudo tc qdisc add dev eth0 root netem delay 200ms loss 2%

# 场景4: 动态延迟（模拟突发）
sudo tc qdisc add dev eth0 root netem delay 100ms 200ms

# 3. 查看配置
sudo tc qdisc show dev eth0

# 4. 清除配置
sudo tc qdisc del dev eth0 root

# 5. 针对特定IP/端口
# 使用 iptables 配合 tc
iptables -t mangle -A OUTPUT -d 192.168.1.100 -j MARK --set-mark 1
tc qdisc add dev eth0 root handle 1: prio
tc qdisc add dev eth0 parent 1:1 handle 10: netem delay 300ms
tc filter add dev eth0 protocol ip parent 1:0 prio 1 handle 1 fw flowid 1:1
```

### 安全注意事项
⚠️ **警告**:
- 在生产环境注入前，必须在测试环境充分验证
- 确保有回滚机制（保存当前 tc 配置）
- 避免在关键业务时段测试
- 设置超时自动恢复：`timeout 300 tc qdisc add ...`

## 验证方法

### 验证步骤
1. **基线测试**: 注入前测量正常延迟
2. **注入延迟**: 应用 tc 规则
3. **验证生效**: 使用 ping/mtr 确认延迟已增加
4. **应用测试**: 运行应用功能测试
5. **性能测试**: 使用 iperf3 测试吞吐量变化
6. **清理环境**: 移除 tc 规则，验证恢复

### 验证工具
```bash
#!/bin/bash
# 验证脚本

echo "=== 1. 基线测试 ==="
ping -c 10 target_host | tail -1

echo -e "\n=== 2. 注入延迟 ==="
sudo tc qdisc add dev eth0 root netem delay 300ms

echo -e "\n=== 3. 验证延迟已生效 ==="
sleep 2
ping -c 10 target_host | tail -1

echo -e "\n=== 4. 性能测试 ==="
iperf3 -c target_host -t 30 -J

echo -e "\n=== 5. 清理环境 ==="
sudo tc qdisc del dev eth0 root

echo -e "\n=== 6. 验证恢复 ==="
sleep 2
ping -c 10 target_host | tail -1
```

### 成功标准
- ✅ ping RTT 增加到预期值（如 300ms）
- ✅ 应用功能仍然可用
- ✅ CUBIC 算法吞吐量明显下降
- ✅ BBR 算法吞吐量保持较好（测试 BBR 优势）
- ✅ 清理 tc 规则后延迟恢复正常

## 预防措施

### 设计层面
1. **CDN 加速**: 静态资源就近访问
2. **多地域部署**: 服务就近接入
3. **连接复用**: HTTP/2, gRPC 多路复用
4. **异步处理**: 避免同步等待
5. **超时设置**: 合理的连接/读写超时

### 实现层面
1. **使用 BBR**: 替代 CUBIC，改善高延迟网络
2. **TCP 快速打开**: TFO 减少握手延迟
3. **应用层优化**: 批量请求、pipeline
4. **协议选择**: UDP 用于实时应用（如 QUIC）

### 运维层面
1. **监控告警**: 实时监控 RTT，设置阈值告警
2. **路由优化**: 选择最优路由
3. **带宽规划**: 预留足够带宽，避免拥塞
4. **定期测试**: 使用 iperf3 定期测试网络性能

## 恢复策略

### 自动恢复
- TCP 超时重传
- 负载均衡自动切换到备用链路
- 路由协议收敛（OSPF/BGP）

### 手动恢复
```bash
# 1. 检查网络状态
mtr -r -c 100 target_host

# 2. 检查路由表
ip route get target_host

# 3. 检查带宽使用
iftop -i eth0

# 4. 流量整形（临时）
sudo tc qdisc add dev eth0 root tbf rate 1gbit burst 32kbit latency 400ms

# 5. 切换到备用链路
ip route add default via <backup_gateway> metric 100
```

### 恢复时间
- 自动恢复：几秒到几分钟（取决于 TCP 超时）
- 手动干预：15-30 分钟

## 测试用例

### 测试场景1: 跨国访问性能
**描述**: 模拟从中国访问美国服务

**步骤**:
```bash
# 设置 300ms 延迟（模拟中美RTT）
sudo tc qdisc add dev eth0 root netem delay 300ms 50ms

# 测试 CUBIC vs BBR
sysctl -w net.ipv4.tcp_congestion_control=cubic
iperf3 -c server -t 60 -J > cubic_result.json

sysctl -w net.ipv4.tcp_congestion_control=bbr
iperf3 -c server -t 60 -J > bbr_result.json

# 对比结果
```

**预期结果**: BBR 吞吐量比 CUBIC 高 30-50%

### 测试场景2: 延迟抖动影响
**描述**: 测试延迟不稳定对应用的影响

**步骤**:
```bash
# 设置 200ms ± 100ms 抖动
sudo tc qdisc add dev eth0 root netem delay 200ms 100ms 25%

# 运行应用测试
# 观察是否有超时、重连
```

**预期结果**: 应用能正常处理抖动，无大量超时

### 测试场景3: 逐渐增加延迟
**描述**: 测试系统在延迟逐渐增加时的表现

**步骤**:
```bash
for delay in 50 100 200 500 1000; do
    echo "Testing with ${delay}ms delay"
    sudo tc qdisc change dev eth0 root netem delay ${delay}ms
    ping -c 50 target_host | tail -1
    iperf3 -c target_host -t 10
    sleep 5
done
```

**预期结果**: 延迟增加时吞吐量逐渐下降，但不会完全失败

## 真实案例

### 案例1: 跨国应用响应慢
**时间**: 2023-08-15
**影响**: 中国用户访问美国服务器，API 响应时间从 100ms 增加到 3s
**原因**: 跨国链路拥塞，RTT 从 180ms 增加到 800ms
**解决**:
1. 启用 BBR 拥塞控制
2. 部署新加坡节点
3. 使用 CDN 加速静态资源
**结果**: API 响应时间降低到 200ms

### 案例2: 云服务延迟问题
**时间**: 2023-11-20
**影响**: 数据库查询超时率从 0.1% 上升到 15%
**原因**: 云服务商网络故障，路由绕行，RTT 从 5ms 增加到 200ms
**解决**:
1. 增加数据库连接超时时间
2. 启用连接池重用
3. 切换到备用可用区
**教训**: 需要配置合理的超时和重试机制

### 案例3: 游戏卡顿问题
**时间**: 2024-01-10
**影响**: 玩家反馈游戏卡顿，延迟从 50ms 上升到 500ms
**原因**: 运营商链路拥塞，晚高峰时段严重
**解决**:
1. 游戏服务器使用 UDP 而非 TCP
2. 实现客户端预测和插值算法
3. 多地部署，智能路由
**结果**: 卡顿率降低 80%
