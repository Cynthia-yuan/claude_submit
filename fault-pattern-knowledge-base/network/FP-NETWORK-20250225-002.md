## 故障描述

### 定义
数据包在传输过程中丢失，导致重传、吞吐量下降、应用错误。

### 典型场景
- 网络拥塞导致队列溢出
- 无线网络干扰
- 物理链路故障
- 路由器/交换机性能不足
- 防火墙丢弃

### 影响范围
- **吞吐量**: TCP 因丢包降速
- **延迟**: 重传增加延迟
- **实时应用**: 语音/视频卡顿、马赛克
- **数据完整性**: UDP 丢包导致数据缺失

## 故障表现

### 症状
- `ping` 显示 packet loss
- TCP 重传率高
- 应用超时或错误
- 视频/音频质量下降
- 文件传输速度慢

## 根本原因

### 常见原因
1. **拥塞丢包**: 队列满，路由器主动丢包（Tail Drop/RED）
2. **链路质量**: 无线干扰、光纤衰减、信号弱
3. **设备故障**: 端口错误、硬件故障
4. **配置问题**: MTU 不匹配导致分片丢失
5. **安全策略**: 防火墙规则丢弃

### 触发条件
- 链路利用率 > 90%
- 无线环境干扰源（微波、蓝牙）
- 劣质网线/光纤
- 设备过载

## 检测方法

### 主动检测
```bash
# 1. Ping 测试
ping -c 1000 -s 1472 target_host  # 大包测试
# 观察丢包率

# 2. MTR
mtr -r -c 500 -s 1472 target_host
# 识别哪一跳开始丢包

# 3. iperf3 测试
iperf3 -c target_host -t 60 -J
# 查看重传: jq '.end.sum_sent.retransmits'

# 4. ethtool 统计
ethtool -S eth0 | grep -i error
# 查看物理层错误

# 5. sar 统计
sar -n EDEV 1 10
# 网络错误统计
```

## 故障注入

### 注入方法
使用 `tc netem` 模拟丢包

### 注入工具
```bash
#!/bin/bash
# 网络丢包注入脚本

# 场景1: 固定丢包率
sudo tc qdisc add dev eth0 root netem loss 2%

# 场景2: 随机丢包（25%相关性）
sudo tc qdisc add dev eth0 root netem loss 2% 25%

# 场景3: 丢包 + 延迟
sudo tc qdisc add dev eth0 root netem delay 200ms loss 1%

# 场景4: 突发丢包（模拟真实网络）
sudo tc qdisc add dev eth0 root netem loss 0.5% 25% \
  delay 100ms 50ms loss gemodel 1% 33% 75%

# 场景5: 定期丢包（每N个包丢一个）
# 需要自定义脚本

# 验证
ping -c 100 target_host
iperf3 -c target_host -t 30

# 清除
sudo tc qdisc del dev eth0 root
```

### 高级：按条件丢包
```bash
# 只对特定端口丢包
iptables -A OUTPUT -p tcp --dport 443 -m statistic --mode random \
  --probability 0.01 -j DROP

# 只对特定IP丢包
iptables -A OUTPUT -d 192.168.1.100 -m statistic --mode random \
  --probability 0.05 -j DROP

# 查看规则
iptables -L -v -n

# 清除
iptables -F OUTPUT
```

### 安全注意事项
⚠️ **警告**:
- 丢包会触发 TCP 重传，可能导致延迟放大
- 测试 UDP 应用时，丢包会导致数据永久丢失
- 避免在关键业务时段测试

## 验证方法

### 验证步骤
1. **基线测试**: 测量正常丢包率和重传率
2. **注入丢包**: 应用 tc 规则
3. **验证生效**: ping/iperf3 确认丢包
4. **应用测试**:
   - TCP 应用：测试重传机制
   - UDP 应用：测试丢包处理
5. **性能对比**: 对比 CUBIC vs BBR
6. **清理环境**: 移除规则

### 验证工具
```bash
#!/bin/bash
# 完整验证脚本

echo "=== 1. 基线测试 ==="
iperf3 -c target_host -t 30 | grep "retransmits"

echo -e "\n=== 2. 注入 2% 丢包 ==="
sudo tc qdisc add dev eth0 root netem loss 2%

echo -e "\n=== 3. 验证丢包 ==="
ping -c 100 target_host | grep "packet loss"

echo -e "\n=== 4. 测试 TCP 重传 ==="
iperf3 -c target_host -t 30 -J | jq '.end.sum_sent.retransmits'

echo -e "\n=== 5. 测试不同拥塞算法 ==="
sysctl -w net.ipv4.tcp_congestion_control=cubic
iperf3 -c target_host -t 30 -J > cubic_loss.json

sysctl -w net.ipv4.tcp_congestion_control=bbr
iperf3 -c target_host -t 30 -J > bbr_loss.json

echo -e "\n=== 对比吞吐量 ==="
echo "CUBIC:" $(jq -r '.end.sum_received.bits_per_second' cubic_loss.json | awk '{print $1/1000000, "Mbps"}')
echo "BBR:" $(jq -r '.end.sum_received.bits_per_second' bbr_loss.json | awk '{print $1/1000000, "Mbps"}')

echo -e "\n=== 6. 清理 ==="
sudo tc qdisc del dev eth0 root
```

### 成功标准
- ✅ ping 显示预期丢包率
- ✅ TCP 重传率增加
- ✅ CUBIC 吞吐量显著下降
- ✅ BBR 吞吐量下降较少（BBR 优势场景）
- ✅ UDP 应用有丢包处理机制（FEC、重传）

## 预防措施

### 设计层面
1. **应用层冗余**: 关键数据发送多份
2. **前向纠错**: FEC（Forward Error Correction）
3. **协议选择**: 实时数据用 UDP + 应用层可靠性
4. **多路径传输**: MPTCP

### 实现层面
1. **使用 BBR**: 不因丢包降速
2. **TCP 快速重传**: 减少等待超时
3. **应用层重试**: 指数退避
4. **数据校验**: 检测数据完整性

### 运维层面
1. **监控丢包率**: 实时告警
2. **网络优化**: 避免拥塞
3. **设备升级**: 及时更换老化设备
4. **链路冗余**: 主备切换

## 恢复策略

### 自动恢复
- TCP 超时重传
- 快速重传机制（3个重复 ACK）
- 负载均衡切换链路

### 手动恢复
```bash
# 1. 诊断丢包位置
mtr -r -c 500 target_host

# 2. 检查链路状态
ethtool -S eth0 | grep error

# 3. 检查队列长度
tc qdisc show dev eth0

# 4. 调整队列
# 启用 fq_codel（Active Queue Management）
sudo tc qdisc replace dev eth0 root fq_codel

# 5. 切换链路
ip route replace default via <backup_gateway>
```

### 恢复时间
- 少量丢包：TCP 自动恢复（< 1s）
- 大量丢包：需要切换链路（5-10 分钟）

## 测试用例

### 测试场景1: CUBIC vs BBR 在丢包环境
**描述**: 验证 BBR 在丢包场景优于 CUBIC

**步骤**:
```bash
# 测试不同丢包率
for loss_rate in 0.1 0.5 1 2 5; do
    echo "Testing with ${loss_rate}% loss"

    sudo tc qdisc add dev eth0 root netem loss ${loss_rate}%

    sysctl -w net.ipv4.tcp_congestion_control=cubic
    cubic_throughput=$(iperf3 -c server -t 30 -J | \
      jq -r '.end.sum_received.bits_per_second')

    sysctl -w net.ipv4.tcp_congestion_control=bbr
    bbr_throughput=$(iperf3 -c server -t 30 -J | \
      jq -r '.end.sum_received.bits_per_second')

    echo "CUBIC: $((cubic_throughput/1000000)) Mbps"
    echo "BBR:   $((bbr_throughput/1000000)) Mbps"

    sudo tc qdisc del dev eth0 root
    sleep 5
done
```

**预期结果**: 丢包率越高，BBR 优势越明显

### 测试场景2: UDP 丢包处理
**描述**: 测试 UDP 应用在丢包环境的表现

**步骤**:
```bash
# 注入丢包
sudo tc qdisc add dev eth0 root netem loss 5%

# 运行 UDP 测试
iperf3 -c server -u -b 100M -t 30 -J

# 观察丢包率
```

**预期结果**: UDP 丢包率接近注入值，应用层应有处理

### 测试场景3: 丢包 + 延迟组合
**描述**: 真实场景通常是丢包和延迟同时存在

**步骤**:
```bash
# 高延迟 + 丢包（卫星链路模拟）
sudo tc qdisc add dev eth0 root netem delay 600ms loss 2%

# 测试视频通话质量
# 测试文件传输
```

**预期结果**: 应用能正常工作，体验可接受

## 真实案例

### 案例1: 跨国视频会议卡顿
**时间**: 2023-06-20
**影响**: 视频会议频繁卡顿、马赛克
**原因**: 国际链路丢包率 3-5%
**解决**:
1. 启用 UDP 协议替代 TCP
2. 实现自适应码率
3. 增加 FEC 冗余
**结果**: 丢包容忍度提高到 10%

### 案例2: 游戏数据包丢失
**时间**: 2023-10-15
**影响**: 玩家位置不同步、瞬移
**原因**: 弱网环境丢包率 > 10%
**解决**:
1. 实现客户端预测
2. 关键操作三次握手确认
3. 位置插值平滑
**结果**: 体验改善 70%

### 案例3: 文件传输缓慢
**时间**: 2024-01-30
**影响**: 文件传输速度只有理论带宽的 20%
**原因**: 链路丢包导致 TCP 降速
**解决**:
1. 启用 BBR
2. 增加窗口大小
3. 使用多线程传输
**结果**: 速度提升 4 倍
