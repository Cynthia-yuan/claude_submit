## 故障描述

### 定义
磁盘 I/O 请求响应时间异常增大，超过正常基线，导致系统性能下降、进程等待。

### 典型场景
- 机械硬盘随机读取
- SSD 写满后的 GC 压力
- RAID 重建/降级
- 磁盘队列堆积
- 后台任务占用 I/O

### 影响范围
- **系统性能**: iowait 升高，CPU 空转
- **应用响应**: 数据库查询慢、文件读写卡顿
- **用户体验**: 操作卡顿、加载缓慢
- **影响层级**: 应用层、文件系统层、块设备层

## 故障表现

### 症状
- `top` 显示 `iowait` 高（> 10%）
- `iostat` 显示 `await` 高（> 20ms）
- 应用日志显示 "I/O timeout"
- 进程处于 `D` 状态（不可中断睡眠）
- 文件操作响应慢

## 根本原因

### 常见原因
1. **磁盘性能瓶颈**: HDD 随机读写慢、SSD 写满
2. **队列堆积**: I/O 请求 > 处理能力
3. **RAID 重建**: RAID 降级后重建占用带宽
4. **后台任务**: 备份、索引、同步占用 I/O
5. **磁盘故障**: 即将失败、坏块扩散
6. **文件系统碎片**: 碎片导致额外寻道

### 触发条件
- 大量随机 I/O（数据库、虚拟机）
- 磁盘空间 > 90%
- RAID 成员故障
- 并发 I/O 突增

## 检测方法

### 主动检测
```bash
# 1. iostat 实时监控
iostat -x 1 10
# 关注: await, svctm, %util

# 2. iotop 找出 I/O 进程
sudo iotop -oP
# -o: 只显示有I/O的进程
# -P: 只显示进程，不显示线程

# 3. pidstat 统计
pidstat -d 1 10
# 每个进程的 I/O 统计

# 4. fio 基准测试
fio --name=randread --filename=/disk/test \
    --rw=randread --bs=4k --numjobs=4 \
    --time_based --runtime=60 --group_reporting

# 5. 检查 D 状态进程
ps aux | awk '$8 ~ /D/'

# 6. 查看块设备统计
cat /sys/block/sda/queue/iosched  # 调度器
cat /proc/diskstats
```

### 被动监控
```yaml
monitoring_tools:
  - tool: "Prometheus + node_exporter"
    metrics: [
      "node_disk_io_time_seconds_total",
      "node_disk_io_time_weighted_seconds",
      "node_disk_read_time_seconds_total",
      "node_disk_write_time_seconds_total"
    ]

  - tool: "eBPF"

      #!/usr/bin/env bpftrace
      # 跟踪 I/O 延迟
      kprobe:submit_bio
      {
        @start[args] = nsecs;

      kretprobe:submit_bio /@start[args]/

    feature: "按 cgroup 查看 I/O"
```

## 故障注入

### 注入方法
使用 `dm-delay` (Device Mapper Delay) 模拟 I/O 延迟

### 注入工具
```bash
#!/bin/bash
# I/O 延迟注入脚本

# 准备：确保加载 dm-delay 模块
sudo modprobe dm-delay

# 创建 I/O 延迟设备
# 原始设备
ORIGINAL_DEV="/dev/sdb"

# 场景1: 固定延迟
# 读写都延迟 100ms
sudo dmsetup create delayed-disk --table "0 $(blockdev --getsize $ORIGINAL_DEV) delay $ORIGINAL_DEV 0 100"

# 场景2: 读写不同延迟
# 读 50ms, 写 200ms
sudo dmsetup create delayed-disk --table "0 $(blockdev --getsize $ORIGINAL_DEV) delay $ORIGINAL_DEV 0 50 $ORIGINAL_DEV 0 200"

# 场景3: 逐渐增加延迟
for delay in 10 50 100 200 500; do
    echo "Setting delay to ${delay}ms"
    sudo dmsetup reload delayed-disk --table "0 $(blockdev --getsize $ORIGINAL_DEV) delay $ORIGINAL_DEV 0 $delay"
    sudo dmsetup resume delayed-disk
    sleep 60
done

# 场景4: 使用 cgroup 限制 I/O
# 创建 cgroup
sudo mkdir -p /sys/fs/cgroup/blkio/test
echo "8:16 1048576" | sudo tee /sys/fs/cgroup/blkio/test/blkio.throttle.read_bps_device
# 限制读取速度为 1MB/s (模拟慢速磁盘)

# 验证
sudo dmsetup ls
sudo dmsetup table delayed-disk
# 使用新设备: /dev/mapper/delayed-disk

# 性能测试
fio --name=test --filename=/dev/mapper/delayed-disk --rw=read --bs=4k --iodepth=32 --runtime=30

# 清除
sudo dmsetup remove delayed-disk
```

### 使用 cgroups v2 注入
```bash
# 挂载 cgroups v2
sudo mount -t cgroup2 none /sys/fs/cgroup

# 创建受限 cgroup
sudo mkdir /sys/fs/cgroup/slow_io

# 限制 I/O 带宽 (模拟慢速磁盘)
echo "8:0 rbps=1048576" | sudo tee /sys/fs/cgroup/slow_io/io.max
# 限制 /dev/sda (8:0) 读取速度为 1MB/s

# 将进程放入受限 cgroup
echo $PID | sudo tee /sys/fs/cgroup/slow_io/cgroup.procs
```

### 安全注意事项
⚠️ **警告**:
- I/O 延迟会导致进程阻塞，可能影响系统稳定性
- 不要在系统盘或关键数据盘上测试
- 确保数据已备份
- 设置超时自动恢复

## 验证方法

### 验证步骤
1. **基线测试**: fio 测试正常 I/O 延迟
2. **注入延迟**: 创建 dm-delay 设备
3. **验证生效**: iostat/iostat 确认延迟增加
4. **应用测试**: 运行应用并观察性能
5. **压力测试**: fio 持续 I/O 压力
6. **清理环境**: 移除 dm-delay 设备

### 验证工具
```bash
#!/bin/bash
# 完整验证脚本

DEVICE="/dev/sdb"
DELAY_MS=100

echo "=== 1. 基线测试 ==="
fio --name=baseline --filename=$DEVICE --rw=read --bs=4k --iodepth=1 --runtime=10 | grep "lat"

echo -e "\n=== 2. 注入延迟 ==="
sudo modprobe dm-delay
sudo dmsetup create delayed-disk --table "0 $(blockdev --getsize $DEVICE) delay $DEVICE 0 $DELAY_MS"

echo -e "\n=== 3. 验证延迟已生效 ==="
sleep 2
fio --name=delayed --filename=/dev/mapper/delayed-disk --rw=read --bs=4k --iodepth=1 --runtime=10 | grep "lat"

echo -e "\n=== 4. 监控系统 I/O ==="
iostat -x 1 5 | grep -E "Device|delayed-disk"

echo -e "\n=== 5. 应用层测试 ==="
# 模拟数据库读取
time dd if=/dev/mapper/delayed-disk of=/dev/null bs=4K count=10000

echo -e "\n=== 6. 清理 ==="
sudo dmsetup remove delayed-disk
```

### 成功标准
- ✅ iostat 显示 await 增加到预期值（如 100ms）
- ✅ fio 测试延迟符合预期
- ✅ 应用能正常处理（可能变慢）
- ✅ 进程出现 D 状态（高延迟时）
- ✅ 清理后延迟恢复正常

## 预防措施

### 设计层面
1. **SSD 替代 HDD**: 降低 99% 的延迟
2. **RAID 10**: 提升读写性能
3. **缓存层**: Redis/Memcached 缓存热点数据
4. **异步 I/O**: 避免阻塞

### 实现层面
1. **I/O 调度器**:
   - HDD: 使用 deadline 或 cfq
   - SSD: 使用 noop 或 mq-deadline
2. **队列深度**: 适当增大队列
3. **批量操作**: 合并小 I/O
4. **连接池**: 减少频繁打开关闭文件

### 运维层面
1. **监控**: 实时监控 iostat
2. **容量规划**: 保持磁盘空间 < 80%
3. **定期维护**: 碎片整理（HDD）、TRIM（SSD）
4. **后台任务**: 低峰期运行

## 恢复策略

### 自动恢复
- SSD GC 完成
- RAID 重建完成
- 后台任务结束

### 手动恢复
```bash
# 1. 诊断问题设备
iostat -x 1 5  # 找出高延迟设备
iotop -oP      # 找出高 I/O 进程

# 2. 限制 I/O 进程
ionice -c 3 -p $PID  # 设置为 idle 优先级
# 或
cpulimit -p $PID -l 20  # 限制 CPU（间接降低 I/O）

# 3. 更换 I/O 调度器
echo mq-deadline | sudo tee /sys/block/sda/queue/scheduler

# 4. 增加队列深度
echo 256 | sudo tee /sys/block/sda/queue/nr_requests

# 5. 清理后台任务
sudo systemctl stop backup.service

# 6. 扩容
# 添加新磁盘到 RAID 或 LVM
```

### 恢复时间
- 自动恢复：几分钟到几小时
- 手动干预：15-60 分钟

## 测试用例

### 测试场景1: 高延迟下的数据库性能
**描述**: 测试 I/O 延迟对数据库的影响

**步骤**:
```bash
# 准备数据库
mysql -e "CREATE DATABASE test_io; USE test_io; \
  CREATE TABLE t (id INT PRIMARY KEY, data VARCHAR(1000)); \
  INSERT INTO t SELECT * FROM generate_series(1,100000);"

# 注入不同延迟
for delay in 10 50 100 200; do
    sudo dmsetup create delayed-disk --table "0 $(blockdev --getsize /dev/sdb) delay /dev/sdb 0 $delay"

    # 测试查询性能
    time mysql -e "SELECT COUNT(*) FROM test_io.t;"

    sudo dmsetup remove delayed-disk
done
```

**预期结果**: 延迟越高，查询越慢

### 测试场景2: 延迟阈值测试
**描述**: 找出应用能容忍的最大 I/O 延迟

**步骤**:
```bash
for delay in 1 5 10 20 50 100 200 500 1000; do
    echo "Testing ${delay}ms delay"

    sudo dmsetup create delayed-disk --table "0 $(blockdev --getsize /dev/sdb) delay /dev/sdb 0 $delay"

    # 运行应用关键操作
    if ! ./run_critical_operation.sh; then
        echo "应用在 ${delay}ms 延迟时失败"
        break
    fi

    sudo dmsetup remove delayed-disk
done
```

**预期结果**: 确定应用的延迟容忍阈值

### 测试场景3: I/O 突发延迟
**描述**: 模拟间歇性的 I/O 延迟

**步骤**:
```bash
# 定时脚本：每30秒注入10秒高延迟
while true; do
    echo "High delay start"
    sudo dmsetup create delayed-disk --table "0 $(blockdev --getsize /dev/sdb) delay /dev/sdb 0 500"
    sleep 10
    sudo dmsetup remove delayed-disk
    echo "Normal delay"
    sleep 30
done
```

**预期结果**: 应用能恢复，不会永久卡住

## 真实案例

### 案例1: 数据库查询超时
**时间**: 2023-07-10
**影响**: 数据库查询超时率从 0.1% 上升到 30%
**原因**: RAID 5 阵列重建，I/O 延迟从 5ms 增加到 200ms
**解决**:
1. 暂停非关键查询
2. 调整重建速度
3. 添加读缓存
**结果**: 超时率降低到 1%

### 案例2: 虚拟机启动慢
**时间**: 2023-11-25
**影响**: 虚拟机启动时间从 30s 增加到 10分钟
**原因**: 存储系统后台数据迁移，I/O 延迟飙升
**解决**:
1. 迁移到其他存储
2. 调整迁移时间窗口
3. 增加 SSD 缓存层
**结果**: 启动时间恢复到 45s

### 案例3: 日志写入导致系统卡顿
**时间**: 2024-02-05
**影响**: 日志服务占用 I/O，系统整体变慢
**原因**: 同步写入日志 + 磁盘慢
**解决**:
1. 改为异步写入
2. 日志缓冲
3. 使用独立磁盘
**结果**: 系统响应恢复正常
