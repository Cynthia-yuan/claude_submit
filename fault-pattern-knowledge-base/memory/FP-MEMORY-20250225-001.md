## 故障描述

### 定义
系统物理内存和交换空间全部耗尽，内核 OOM Killer 被触发，强制终止进程以释放内存。

### 典型场景
- 内存泄漏
- 突发流量导致内存暴涨
- 配置不当（超卖）
- 大文件处理
- 并发数过多

### 影响范围
- **服务中断**: 进程被 OOM Killer 杀死
- **数据丢失**: 未保存的数据丢失
- **系统不稳定**: 关键进程可能被杀
- **影响层级**: 系统级、应用级

## 故障表现

### 症状
- 系统卡顿，响应极慢
- 进程突然消失（被 OOM Killer 杀死）
- `dmesg` 显示 "Out of memory"
- Swap 使用率 100%
- `free` 显示可用内存接近 0

## 根本原因

### 常见原因
1. **内存泄漏**: 程序 bug，未释放内存
2. **配置错误**: JVM 堆内存设置过大
3. **流量突增**: 缓存、连接数暴涨
4. **资源超卖**: 容器/虚拟机内存总和 > 物理内存
5. **大文件处理**: 一次性加载大文件到内存
6. **并发增加**: 并发任务数超出预期

### 触发条件
- `MemAvailable + SwapFree < 申请内存`
- `/proc/sys/vm/min_free_kbytes` 阈值触发
- `vm.panic_on_oom = 1` 时可能 panic

## 检测方法

### 主动检测
```bash
# 1. 检查内存使用
free -h
# 关注: available, swap used

# 2. 检查进程内存
ps aux --sort=-%mem | head -20
# 找出内存占用最高的进程

# 3. 检查 OOM 日志
dmesg | grep -i "out of memory"
dmesg | grep -i "killed process"
journalctl -k | grep -i oom

# 4. 检查 OOM 统计
cat /proc/vmstat | grep oom

# 5. 检查 swap 使用
cat /proc/swaps
vmstat 1 5  # si/so 列表示 swap in/out

# 6. slab 内存占用
cat /proc/meminfo | grep -i slab
slabtop  # 实时查看

# 7. 内存映射
pmap $PID  # 进程内存映射
smem      # 内存使用分析

# 8. cgroup 内存限制
cat /sys/fs/cgroup/memory/memory.limit_in_bytes
cat /sys/fs/cgroup/memory/memory.usage_in_bytes
```

## 故障注入

### 注入方法
使用 `stress-ng`, `malloc`, 或 cgroup 限制触发 OOM

### 注入工具
```bash
#!/bin/bash
# 内存耗尽注入脚本

# 方法1: 使用 stress-ng
# 快速消耗内存
stress-ng --vm 1 --vm-bytes $(awk '/MemAvailable/{printf "%d\n", $2 * 0.9}' <(free -m))M --timeout 60s

# 逐渐消耗内存（缓慢泄漏模拟）
stress-ng --vm 1 --vm-bytes $(awk '/MemAvailable/{printf "%d\n", $2 * 0.5}' <(free -m))M --vm-method all --timeout 300s

# 方法2: 使用 malloc 程序
cat > eat_memory.c << 'EOF'
#include <stdlib.h>
#include <stdio.h>
#include <unistd.h>

int main() {
    size_t chunk_size = 100 * 1024 * 1024;  // 100MB per chunk
    void *ptr;
    int count = 0;

    while (1) {
        ptr = malloc(chunk_size);
        if (ptr == NULL) {
            printf("Failed at %d chunks (%.2f GB)\n",
                   count, count * chunk_size / 1024.0 / 1024 / 1024);
            break;
        }
        // 实际写入内存（不仅是分配）
        memset(ptr, 0, chunk_size);
        count++;
        printf("Allocated %d chunks (%.2f GB)\n",
               count, count * chunk_size / 1024.0 / 1024 / 1024);
        sleep(1);
    }

    // 保持占用
    sleep(3600);
    return 0;
}
EOF

gcc -o eat_memory eat_memory.c
./eat_memory

# 方法3: 使用 cgroup v2 限制
# 创建受限 cgroup
sudo mkdir /sys/fs/cgroup/memory_limited

# 设置内存限制（例如：500MB）
echo "500M" | sudo tee /sys/fs/cgroup/memory_limited/memory.max

# 设置 swap 限制
echo "0" | sudo tee /sys/fs/cgroup/memory_limited/memory.swap.max

# 将进程放入受限 cgroup
echo $PID | sudo tee /sys/fs/cgroup/memory_limited/cgroup.procs

# 方法4: Python 脚本模拟内存泄漏
python3 << 'EOF'
import time
import gc

leaked_memory = []

while True:
    # 每次分配 10MB
    chunk = bytearray(10 * 1024 * 1024)
    leaked_memory.append(chunk)
    print(f"Leaked {len(leaked_memory) * 10} MB")
    time.sleep(1)
EOF

# 方法5: 在容器中触发
docker run -m 512m --memory-swap 512m \
  --oom-kill-disable \
  ubuntu:latest \
  /bin/bash -c "tail /dev/zero"
```

### 安全注意事项
⚠️ **警告**:
- OOM 会杀死进程，可能导致服务中断
- 关键进程（如数据库）被杀会导致数据不一致
- 避免在生产环境直接测试
- 确保有数据备份
- 准备好恢复脚本

## 验证方法

### 验证步骤
1. **基线检查**: `free -h` 查看当前内存
2. **限制内存**: 设置 cgroup 或容器限制
3. **触发 OOM**: 运行内存消耗程序
4. **观察行为**:
   - Swap 开始使用
   - 系统变慢
   - OOM Killer 触发
5. **检查日志**: `dmesg` 确认进程被杀
6. **验证恢复**: 进程重启后恢复

### 验证工具
```bash
#!/bin/bash
# 完整验证脚本

echo "=== 1. 基线检查 ==="
free -h
cat /proc/swaps

echo -e "\n=== 2. 创建内存受限环境 ==="
LIMIT=512M
CGROUP=/sys/fs/cgroup/memory_test
sudo mkdir -p $CGROUP
echo $LIMIT | sudo tee $CGROUP/memory.max
echo 0 | sudo tee $CGROUP/memory.swap.max

echo -e "\n=== 3. 启动测试进程 ==="
python3 << 'PYEOF' &
test_pid=$!
import time
chunks = []
while True:
    chunks.append(bytearray(100 * 1024 * 1024))  # 100MB
    print(f"Allocated {len(chunks) * 100} MB")
    time.sleep(1)
PYEOF

# 将进程放入 cgroup
echo $test_pid | sudo tee $CGROUP/cgroup.procs

echo -e "\n=== 4. 监控内存使用 ==="
watch -n 1 "echo '=== Memory ===' && free -h && echo '=== Cgroup ===' && cat $CGROUP/memory.current" &
watch_pid=$!

echo -e "\n=== 5. 等待 OOM ==="
timeout 120 tail --pid=$test_pid -f /dev/null

# 检查结果
echo -e "\n=== 6. 检查 OOM 日志 ==="
dmesg | tail -20 | grep -i oom

echo -e "\n=== 7. 清理 ==="
kill $watch_pid 2>/dev/null
sudo rmdir $CGROUP
```

### 成功标准
- ✅ Swap 使用率上升
- ✅ `/proc/sys/vm/drop_caches` 清理后被立即使用
- ✅ dmesg 显示 "Out of memory" 和 "Killed process"
- ✅ 进程被终止
- ✅ 系统仍然运行（kernel panic 除外）

## 预防措施

### 设计层面
1. **内存限制**: 容器/进程设置内存上限
2. **优雅降级**: 内存不足时降级服务
3. **资源隔离**: 关键进程独立部署
4. **熔断机制**: 内存超限后拒绝请求

### 实现层面
1. **内存管理**:
   - 及时释放不再使用的内存
   - 避免内存泄漏
   - 使用对象池
2. **监控告警**: 内存使用 > 70% 告警
3. **自动重启**: 检测到内存泄漏后重启
4. **缓存策略**: LRU 淘汰，限制大小

### 运维层面
1. **容量规划**: 预留 20% 内存
2. **监控**: 实时监控内存趋势
3. **定期检查**: 使用 valgrind 检测泄漏
4. **压力测试**: 定期进行 OOM 测试

## 恢复策略

### 自动恢复
- systemd 自动重启服务
- Kubernetes 自动重启 Pod
- OOM Killer 杀死进程释放内存

### 手动恢复
```bash
# 1. 检查被杀进程
dmesg | grep -i "killed process"
journalctl -k | grep -i oom

# 2. 释放缓存（谨慎使用）
echo 3 | sudo tee /proc/sys/vm/drop_caches
# 1: pagecache
# 2: slab objects
# 3: all

# 3. 停止非关键进程
sudo systemctl stop docker
sudo systemctl stop mysql

# 4. 增加 swap（临时）
sudo dd if=/dev/zero of=/swapfile bs=1G count=4
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# 5. 调整 OOM Killer 行为
# 保护关键进程
echo -500 | sudo tee /proc/12345/oom_score_adj  # 降低被杀概率

# 6. 重启服务
sudo systemctl restart critical-service

# 7. 查找内存泄漏
# Java heap dump
jmap -dump:format=b,file=heap.hprof <pid>
# 分析工具: Eclipse MAT, jvisualvm

# C/C++
valgrind --leak-check=full --show-leak-kinds=all ./program
```

### 恢复时间
- 自动重启：1-5 分钟
- 手动恢复：15-30 分钟

## 真实案例

### 案例1: Java 应用 OOM
**时间**: 2023-08-20
**影响**: 核心服务每隔 2 小时重启
**原因**: 内存泄漏，Connection 未关闭
**解决**:
1. heap dump 分析找到泄漏点
2. 修复代码bug
3. 添加内存监控
**结果**: 内存稳定，不再重启

### 案例2: 容器 OOM 导致 Pod 频繁重启
**时间**: 2023-12-15
**影响**: Pod 频繁重启，服务不可用
**原因**: 内存限制设置过小，实际需求 1GB，限制 512MB
**解决**:
1. 分析实际内存使用
2. 调整限制为 1.5GB
3. 添加 HPA（水平Pod自动扩缩容）
**结果**: Pod 稳定运行

### 案例3: 系统级 OOM
**时间**: 2024-01-30
**影响**: 整个系统卡顿，SSH 无法登录
**原因**: 多个容器超卖，总内存 > 物理内存
**解决**:
1. 物理登录
2. 停止非关键容器
3. 配置 kubernetes resource quotas
**结果**: 系统稳定
